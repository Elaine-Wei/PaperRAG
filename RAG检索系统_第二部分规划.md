# AI 论文 RAG 系统 —— 第二部分规划

> 检索层设计与落地路线
> AI Agent 组 · 第二阶段
> 更新日期:2026-07

---

## 一、背景与定位

第一部分(数据采集 + 增强)已完成并跑通:

- arXiv API 抓取 AI / Agent 方向最新论文,入库 PostgreSQL
- 从 Semantic Scholar、Hugging Face Papers 两个来源异步补充 arXiv 没有的字段(引用数、影响力引用数、参考文献数、社区点赞数等)
- Papers with Code 已在 2025 年 7 月被 Meta 关停,已从数据源中移除;其代码链接数据后续可从 Hugging Face 的 `linkedModels` / `linkedSpaces` 补回

第二部分的目标:**在已有论文数据之上,建立一套完整的检索系统**,让用户能够按不同需求从论文库中精准获取内容。

核心设计原则:**统一底座,分层填充,共用一张表一套接口**。检索能力分为 A–F 六个层级,层层叠加,不是六套独立系统。

---

## 二、检索能力的六个层级(A–F)

| 层级 | 能力 | 解决的问题 | 依赖的数据 |
|------|------|-----------|-----------|
| **A** | 摘要级语义检索 | "有没有讲 X 的论文" —— 定位到整篇 | 摘要(第一部分已有) |
| **B** | 段落级语义检索 | "这篇论文的方法具体怎么做" —— 定位到段落 | PDF 全文 |
| **C** | 章节级语义检索 | "跨多篇论文,综合某个技术方向" —— 按章节抽取 | PDF 全文 + 章节结构 |
| **D** | 元数据过滤检索 | "2024 年后、引用 > 50、带代码的论文里……" | 第一部分的增强字段 |
| **E** | 混合检索 + 重排 | 补语义检索短板(精确术语、论文名) | 原始文本 + BM25 + reranker |
| **F** | 引用网络检索(GraphRAG) | "这个方法从哪演化来、被谁改进" —— 沿引用链推理 | 引用关系图 |

### 逐层说明

**A —— 摘要级语义检索**
最简单的一层。摘要在第一部分已经抓到,只需做 embedding 存入向量库。粒度是"一篇论文 = 一个向量"。适合"找论文"的需求。

**B —— 段落级语义检索**
需要下载 PDF、解析全文、切成段落块,每块单独 embedding。粒度是"一篇论文 = 几十个块"。适合"问某篇论文的具体细节"。

**C —— 章节级语义检索**
在 B 的基础上,识别每个块属于哪个章节(intro / method / experiment / conclusion),打上标签。适合"跨多篇论文综合某个特定方面"(如只对比各篇的方法部分)。

**D —— 元数据过滤检索**
在语义检索之上叠加结构化字段过滤。用第一部分攒下的 `published`、`citation_count`、`code_urls` 等字段,先缩小候选范围再做语义匹配。第一部分数据增强的价值在这一层兑现。

**E —— 混合检索 + 重排**
A–D 只用了"向量检索"一条腿。E 加上第二条腿(BM25 关键词检索),两路结果用 RRF 融合,再用 reranker(如 `bge-reranker-v2-m3`)精排。补足向量检索对精确术语、专有名词、论文名的短板。**这一层是检索质量的分水岭。**

**F —— 引用网络检索(GraphRAG)**
最复杂,也是天花板。把论文当成互相连接的网络而非孤立文本块,用引用关系做图上推理。工程量最大(需要图结构),但正是"最前沿"方向所指。可作为长期目标。

### 层级依赖关系

```
A  摘要级语义检索        ┐
B  段落级语义检索        ├─ 语义相似度(一条腿)
C  章节级语义检索        ┘
D  + 元数据字段过滤      ← 用第一部分攒的字段
E  + BM25 + reranker     ← 检索质量分水岭
F  + 引用网络 / GraphRAG ← 最前沿,天花板
```

越往下越依赖前面每一步攒的数据:D 靠增强字段,F 靠引用关系。这也说明第一部分抓取 citation / reference / code 字段并非多余,而是为 D 和 F 铺路。

## 三、技术选型(初步)

| 环节 | 选择 | 理由 | 备选 |
|------|------|------|------|
| 向量库 | PostgreSQL + pgvector | 复用已有数据库,一库同时管结构化数据与向量 | Qdrant / Milvus(规模大后再评估) |
| Embedding 模型 | OpenAI `text-embedding-3-small` | 已有可用 API key;成本极低(约 $0.02 / 百万 token);跑通快 | BGE 系列(需本地部署,后续可省钱/离线时切换) |
| PDF 解析 | 待第二步选型 | —— | PyMuPDF / GROBID(学术论文结构化解析) |
| Reranker | `bge-reranker-v2-m3` | 中英文混合,开源可本地跑 | LLM Reranker(效果更好但需 GPU) |

> 说明:选型原则是"当前规模和阶段下,工程复杂度最低、效果足够好",非绝对最优。规模上来后 D/E/F 相关组件均可重新评估。

## 四、落地路线

底座一次设计到位(支持到 E),能力按 A→B→C→D→E→F 逐层填充,避免返工。

### 里程碑

**第一步:搭底座 + 填 A**
- 建 pgvector 扩展与统一检索表(设计见第五节)
- 摘要 embedding 入库
- 跑通"embedding → 入库 → 检索"最小链路
- 用已有摘要数据,不碰 PDF,最快见效

**第二步:填 B**
- PDF 下载 + 解析 + 段落切块(递归切 + 512 token + 重叠)
- 切块 embedding 存入同表,`granularity = chunk`
- 专注解决 PDF 解析的坑(双栏、公式、扫描版)
- 同步建立图表库:表格结构化(档位2)、图存原图、caption 摘出并记位置指针
- 交叉引用解析:带编号引用(Figure/Table/图/表 + 数字)链接到图表库,记录字符位置

**第三步:填 C**
- 解析时识别章节结构,给块打 `section` 标签
- 在 B 的解析代码上扩展,非重写

**第四步:填 D**
- 检索接口叠加元数据过滤(关联第一部分的 papers 表)

**第五步:填 E**
- 接入 BM25 关键词检索
- RRF 融合向量 + BM25 结果
- reranker 精排
- 检索质量的关键提升

**第六步(长期):填 F**
- 建引用关系图(单独的边表或图数据库)
- 图上检索 / 推理
- 最前沿方向

## 五、切块策略与论文内容的三分架构

A–F 讲的是"检索什么粒度",本节讲"论文原文具体怎么被拆开、存成什么"。这是 B/C 层的前置基础,也决定了后面统一表怎么设计。

核心结论:**一篇论文不是一种内容,而是被拆成三种内容,分别处理。**

```
一篇论文 ─┬─ 正文文本块        → 文本库(切块 + embedding)
          ├─ 表格             → 图表库(结构化数据 + caption + 位置)
          └─ 图 / 可视化       → 图表库(原图 + caption + 位置)
                                    ↑
          正文里的"如图3/表2" ──┘  交叉引用链接,按钮触发悬浮
```

### 5.1 正文文本的切块

**切块方式:递归切(recursive)。** 先按段落切,段落过长再按句子切,仍过长才按固定长度切。兼顾语义完整与块大小可控,是学术结构化文本的常用折中。

**块大小:约 512 token。** 经过验证的甜点区间。太小(如 128)检索精准但缺上下文;太大(如 1024)上下文丰富但检索粗、更费 token。

**块重叠(overlap):要。** 相邻块共享一小段(如 50 token),避免关键信息横跨块边界时两块都不完整。代价是重叠部分被 embedding 两次,可接受。

**参考文献列表:从切块中丢弃。** 论文末尾的引用列表对语义检索是噪音。注意是"从切块里丢",不是"从数据里丢"—— 引用关系要保留给 F 层构图用。

### 5.2 图表脱离正文,单独建"图表库"

一篇论文里所有表格和图,**不切进正文文本块**,而是抽出来单独存为图表库中的独立记录,可按需单独调用。这样正文保持连续干净(不被图表打断而切乱),图表又能被精确定位调用。

每条图表记录包含:

- **原始内容**:表格 = 专业工具提取的结构化数据;图 = **原始图片本身**
- **caption**:图表的说明文字(见 5.4 的特殊处理)
- **位置标记**:在原文中的精确位置(给交叉引用与悬浮用)

**表格处理定为"档位 2":** 用专业表格解析工具(pdfplumber / Camelot / table-transformer)提取行列结构,连同其 caption 作为一个带位置标记的独立单元存出,不混进正文块。

> 表格结构化的更高档位(转成可语义检索的描述文字)列为未来增强,第一版不做。

### 5.3 图(visualization)的特殊性:必须存原图

图和表本质不同:**表格能解析成文字/数据,但图的核心信息在像素里,无法无损转成文字。** 因此:

- **图必须存原始图片本身**,caption 只是辅助。
- 图的清晰度是第一优先级 —— 系统的职责是"精准定位 + 高清呈现",不是"代替人理解图"。

**关于"要不要让 AI 理解图片"——明确不做(设计决策):**

一张图在系统里其实被三处文字描述着:正文的分析("如图3,方法在后期超过基线")、caption 小字、以及图本身。前两者**都是文字,已进文本库、已被 embedding**。所以"这张图在讲什么"靠正文 + caption 已可检索到,**无需再用多模态模型去"看懂"图片**——那样成本高、收益边际,偏离重点。

- **图的检索:靠 caption(+ 正文引用)定位。** 即"定位到我需要的那张图",而非理解图内容。
- **边界说明:** 少数图正文只一笔带过("结果见图5"),信息全在图里。这类图靠文字确实检索不全 —— 但这恰恰印证"原图高清呈现"为何是第一优先级:系统不理解图,只负责把图精准端到人面前,让人自己看。
- 多模态图像理解列为**未来增强(档位 B)**,非第一版目标。

### 5.4 caption 的处理:摘出正文流,保留位置指针

caption 在物理上夹在正文中间,若留在正文里会切断段落;但它逻辑上属于图表而非周围正文。处理方式是把"物理位置"与"逻辑归属"分开:

- **从正文文本流中摘出** caption,使正文段落保持连续、不被切断。
- **保留一个位置指针**(记录它原本在原文的位置),供交叉引用与悬浮插件使用。
- caption 的**逻辑归属是图表**,跟着图表库记录走。
- caption **同时进入检索**(两边用):它既跟着图表存,其文本本身也做 embedding。这样"搜到这张图"既能靠正文提及、也能靠 caption 自身命中,几乎零成本提升检索质量。

### 5.5 交叉引用解析(cross-reference resolution)

识别正文中对图表的指代,建立到图表库对应记录的链接。这是"正文↔图表"缝合的关键。

- **第一版覆盖范围:带编号的显式引用** —— `Figure 3` / `Fig. 3` / `图3` / `Table 2` / `表 2` 等。这类格式规整,正则即可稳定抓取。
- **已知难点(第一版可暂缓):无编号的相对指代** —— "如下图""见上表""the figure above",需靠位置推断(向上/向下找最近的图表),留待后续。
- **链接双向存储:** 正文解释段落 ←→ 图表记录。正向用于"读正文时浮出图";反向用于"直接搜到图后,回看正文如何分析它"。既然解析已在做,顺手记反向,成本低价值翻倍。
- **记录引用的字符位置:** 悬浮插件需要在正文"图3"那几个字上精确挂载,因此解析阶段必须记录每处引用的字符位置。此信息只在解析原文当下可得,切块后丢失便无法补回 —— 属于"当下不做后面补不回"的细节。

### 5.6 悬浮插件(交互功能,依赖上述数据)

面向"论文读得很多"的实际阅读痛点:正文读到"如表2所示",不必滚回去找。

- 正文的图表引用处挂一个按钮,点击即**悬浮**出对应图表:附上**原图** + caption + 分析结论。
- 数据来源:图表库(原图 + caption)+ 交叉引用链接 + 字符位置锚点。
- 归属:属于对外接口 / 前端交互层(见后续接口设计),此处记录其设计意图与数据依赖。
- **依赖前置:** 悬浮插件本身不产生新数据,完全由 5.2–5.5 的后端数据驱动。这些后端数据(图表库、原图、交叉引用、位置锚点)属于"解析当下必须采集、事后补不回"的信息,是本节的重点。

### 5.7 本节对统一表设计的影响

论文被拆成三种内容,意味着存储不是"一张文本表"能装下的:

- 正文文本块 → 进统一检索表(下一节)
- 表格、图 → 需要独立的**图表库**(存结构化数据 / 原图 / caption / 位置)
- 交叉引用链接 → 需要记录"正文位置 ↔ 图表记录"的对应关系

下一节的统一表主要承载**正文文本块**;图表库与交叉引用作为配套结构,与文本表通过 `paper_id` 和位置信息关联。

## 六、embedding 的存储与更新策略

embedding 不是一次性任务。爬虫每天新增论文,模型可能更换,请求会失败 —— 本节设计的是让 embedding **作为一项持续运转的机制**长期稳定运行,而非"跑一次就完"。

### 6.1 增量 embedding:新论文自动纳入

爬虫每天向 `papers` 表写入新论文,这些论文的 embedding 必须自动完成,不能靠手动触发。

做法沿用第一部分已验证的"状态字段驱动"模式:

- 加一个状态字段 **`embedded`(布尔,默认 FALSE)**。
- embedding 任务每次运行,只捞 `embedded = FALSE` 的记录处理,做完置 `TRUE`。
- 天然增量:不会重复做已完成的,也不会漏掉新增的。

这与第一部分 `enriched` 字段的机制完全一致,是同一套可靠模式的复用。

### 6.2 换模型:从"推倒重来"变成"有序迁移"

这是 embedding 长期运转中最棘手的一环,必须在表设计阶段就防好。

**问题本质:** 不同 embedding 模型产出的向量位于**不同的向量空间**,不可混用。拿新模型的查询向量去匹配旧模型产出的库向量,结果是纯噪音。因此一旦更换模型(如从 OpenAI 换到 BGE,或升级到新版模型),**所有既有 embedding 全部作废,必须重做**。对几万篇论文、加全文切块可能几十万个块而言,这是实打实的时间与成本。

**设计对策:记录每条 embedding 的来源模型。**

- 加字段 **`embedding_model`**(如 `text-embedding-3-small`),记录该条向量由哪个模型/版本产出。
- 换模型时,不必一刀切清空全库,而是筛出 `embedding_model != 当前模型` 的记录**分批重跑**。
- 效果:换模型从"灾难性全量重建"降级为"有序、可控、可分批的迁移"。

> 代价仅是每条多存一个字符串字段 —— 廉价的保险。
>
> 更进阶的"同表并存多模型向量以平滑过渡"方案会显著增加表复杂度,第一版不做;只做"记录模型 + 需要时按模型筛选重跑",已足够。

### 6.3 失败处理:自愈式重试

调用 embedding 接口(OpenAI API)会因网络、限流、偶发错误而失败,与第一部分遇到的 Semantic Scholar 429 同理。一批处理到一半某条失败,既不能让整批崩溃,也不能把失败的误标为成功。

做法沿用第一部分 enrichers 的防御模式:

- 每条 / 每批请求 **try/except 包裹,失败打印警告、不中断整批**。
- 失败的记录**保持 `embedded = FALSE`**(绝不误标成功)。
- 下次任务运行时自动重新捞到、重试 —— 失败是**自愈**的,无需额外的重试队列。

### 6.4 批处理:动态攒批 + 分层调度

embedding 接口支持一次传多条文本,批量比逐条调用更快、更省、更不易触发限流。但**批大小不按 A–F 层级硬分,而按文本长度动态决定**。

**为什么不按层级分批大小:** 接口的批量上限卡的是**一批的总 token 数**,不是条数。同为正文块,有的 300 token、有的 800 token,固定条数会导致有的批撑爆、有的没吃满。正确做法是**按总 token 数动态攒批** —— 短文本(如摘要、caption)自动攒成大批,长文本(正文块)自动攒成小批,每批总量顶到接口上限附近。一套逻辑自动适配所有粒度,无需为每层定制。

**各层级的真实 embedding 负载(厘清"强度差异"):**

| 层级 | 是否产生 embedding | 量级 |
|------|:---:|------|
| A 摘要 | 是 | 一篇 = 1 条,几万篇 = 几万条 |
| B 段落 | 是 | 一篇 = 几十块,几万篇 = 几十万~上百万条 |
| C 章节 | 否(给 B 的块加标签,不新增大量块) | 与 B 同量级 |
| D 过滤 | **否**(查询时字段过滤,不产生 embedding) | —— |
| E 混合 | **否**(BM25 是关键词统计;reranker 查询时算少量候选) | —— |
| F 图谱 | **否**(构图,非 embedding 任务) | —— |

**关键结论:真正的大批量 embedding 只发生在 A 和 B/C;D、E、F 几乎不产生 embedding 负载。** 因此"强度不同"的正确应对不是分层定批大小,而是**分层定调度频率**:

- **A(摘要)**:量小、是"找论文"的第一入口、随爬虫每天新增 —— **高频,随爬虫每日增量跑**。
- **B/C(全文块)**:量巨大、成本高 —— **低频或按需触发**(如仅对热门 / 用户可能深挖的论文优先做全文,冷门论文按需),不必每日全量重跑。

**优化项(标注,非第一版):** OpenAI 提供异步、半价的 **Batch API**,适合 B/C 的大批量场景。第一版先用普通同步批量,B/C 量真正上来后再引入 Batch API。

### 6.5 本节对统一表设计的影响

本节新增两个必须进表的字段:

- **`embedded`**(布尔)—— 驱动增量与自愈重试
- **`embedding_model`**(字符串)—— 支撑有序换模型迁移

这两个字段与第一部分 `papers` 表的 `enriched` / `enriched_at` 属同一设计哲学:用状态字段驱动可重入、可恢复的持续任务。

## 七、查询理解(query understanding)

前面几节解决"论文怎么存、怎么被检索到";本节解决检索的**入口**:用户嘴里说出的问题,往往不是适合直接拿去检索的形式,中间需要一层加工。

核心设计:**一次 LLM"意图理解"调用,派生四个产物**,而非多个割裂步骤。

### 7.1 为什么需要这一层(三个会翻车的场景)

- **问题太口语 / 太泛**:"最近 agent 有啥进展" —— 直接检索又杂又浅,"最近"多近、"进展"何指,系统无从抓取信号。
- **中英 / 术语不匹配**:用户中文问"怎么让模型自己规划任务",论文写的是 "task planning / autonomous planning"。向量检索能扛一部分,但 E 层 BM25 靠字面匹配,会完全错过。
- **一个问题含多个意图**:"ReAct 和 Reflexion 在记忆机制上有何区别" —— 实为两篇论文 × 一个特定方面 × 一个对比意图,整句检索不如拆开分别检索再对比。

### 7.2 一次调用,四个产物

query understanding 的核心是一次 LLM 调用,输入用户原始问题,输出一个结构化结果:

```
输入: 用户原始问题
输出: {
  意图类型:    A / B / C ...          → 决定走哪层检索(= 路由)
  是否需要分解: 是 / 否                → 简单问题否,复杂问题是
  查询树:      [叶子查询1, 叶子查询2, ...]  → 分解结果(见 7.4)
  英文术语扩展: [term1, term2, ...]     → 双语检索刚需
}
```

这四个产物**同源** —— 都来自"搞懂用户到底想要什么"这一件事,因此合并在一次调用里最经济,也最自洽。

### 7.3 改写与双语扩展(合并进这一次调用)

- **查询改写**:把口语、模糊的问题规整成检索友好的形式("最近 agent 有啥进展" → "2025 年 agent 领域的新方法与技术进展")。第一版做**轻量改写**(一次 LLM 调用即可),multi-query 式的重度改写留作后续。
- **双语 / 术语扩展**:为查询中的关键概念补上英文术语与同义词("让模型自己规划任务" → 补 `task planning, autonomous planning, agent planning`)。这是本项目**中英混合场景的刚需** —— 向量检索因此更聚焦,BM25(E 层)因此才能字面命中英文论文。
- 两者**由同一次 LLM 调用产出**,不额外增加调用成本。

### 7.4 意图路由:检索的副产物,而非独立步骤

路由(判断走 A / B / C / 是否加 D 过滤)**内嵌于** 7.2 的意图理解,不是一个额外的判断步骤 —— LLM 在理解问题时已经知道用户想要什么粒度:

- "有没有讲 X 的论文" → 意图 A(找整篇)→ 走摘要级
- "ReAct 里 prompt 怎么设计的" → 意图 B(问细节)→ 走段落级
- "2024 年后引用 > 50 的 agent 论文" → 触发 D(元数据过滤)
- "跨多篇对比某方面" → 意图 C → 走章节级 + 触发分解(7.5)

> **实现时序:** 路由的前提是 A/B/C/D 都已建好。按落地路线第一版只有 A,**无多路可选时路由无意义**。因此路由"设计上保留、实现上等 B/C/D 就位后启用";第一版可固定走 A 或加最简规则。

### 7.5 复杂查询的树状分解(query decomposition)

对明显复杂的问题(多意图 / 对比 / 综合),把它分解成一棵**查询树**,叶子是干净、具体、适合检索的子查询,分别检索后按树结构汇总。保留树结构的好处:生成答案时能明确"哪个结果回答哪个子问题",有条理地组织。

```
"ReAct 和 Reflexion 在记忆和推理上谁更强"
   ├─ ReAct 的机制
   │    ├─ ReAct 的记忆机制
   │    └─ ReAct 的推理机制
   └─ Reflexion 的机制
        ├─ Reflexion 的记忆机制
        └─ Reflexion 的推理机制
```

**成本控制:三道闸(关键)。** 分解的成本是乘法 —— 检索次数 = 叶子数,不封顶则单次查询可能裂出十几次检索,又慢又贵,且叶子过窄反伤检索质量。因此:

1. **先判断是否分解** —— 大多数问题是简单的("X 是什么""有没有讲 Y 的论文"),**不分解**,单查询直接检索(零额外成本)。只有多意图 / 对比 / 综合类才进分解流程。
2. **树高上限 2 层** —— 第 1 层拆独立意图,第 2 层拆具体子查询,**不设第 3 层**。论文检索场景 2 层足以覆盖;更深则叶子过窄、检索到的语义不完整。
3. **叶子总数上限 6 个** —— 比树高更该卡的是最终叶子数(决定成本的是"几个检索查询",而非层数)。无论如何分解,单次查询的检索子查询**不超过 6 个**,单次最坏成本因此可预算。

给 LLM 的分解指令即:"把问题分解成检索查询,最多 2 层、总共不超过 6 个;若问题简单则不分解。"

> **实现时序:** 树状分解真正发挥价值需 B/C 就位(简单问题都在 A,无从分解)。第一版**保留设计与接口,暂不实现**;意图理解调用先产出"改写 + 英文扩展 + 路由",分解能力待 B/C 上线后启用。

### 7.6 第一版落地范围小结

| 能力 | 第一版 | 依赖 |
|------|:---:|------|
| 轻量改写 | ✅ 做 | 无 |
| 双语 / 术语扩展 | ✅ 做(刚需) | 无 |
| 意图路由 | 设计保留,最简规则或固定走 A | 需 B/C/D 就位后启用 |
| 树状分解 | 留接口,暂不实现 | 需 B/C 就位 + 三道闸成本控制 |

本节不新增数据表字段 —— query understanding 是**查询时的处理逻辑**,可复用第一部分已有的 LLM 调用能力。可选:将每次查询的理解结果(意图、改写、分解)记入类似 `search_logs` 的日志,供后续检索质量分析(呼应最初蓝图中的 `search_logs` 表)。

## 八、统一表设计要点

一张 pgvector 表承载**正文文本块**,覆盖 A–E,靠字段区分粒度与来源。关键字段:

- **`granularity`** —— 区分粒度:`abstract` / `chunk` / `section`,支撑 A/B/C
- **原始文本字段** —— 不只存向量,同时存原文块。**E 层 BM25 需要原始文本,不是向量**;只存向量则 E 无法实现
- **论文元数据关联** —— 关联第一部分的 `papers` 表(或冗余关键字段),支撑 D 的过滤
- **embedding 向量字段** —— `vector(1536)`,加 `ivfflat` 索引
- **章节标签字段** —— 支撑 C
- **`embedded`(布尔)** —— 驱动增量 embedding 与失败自愈重试(见第六节)
- **`embedding_model`(字符串)** —— 记录向量来源模型,支撑有序换模型迁移(见第六节)

配套结构(见第五节的三分架构):

- **图表库表** —— 存表格结构化数据 / 图原图 / caption / 位置标记,通过 `paper_id` 关联正文
- **交叉引用表** —— 记录"正文字符位置 ↔ 图表记录"的双向对应,驱动悬浮插件
- caption 文本亦做 embedding,可并入统一表(标记来源为 caption)或存于图表库,两边可检索

> F 层的引用图需要额外的边表,建立在同一批论文数据上,不与本表冲突,后续单独设计。

**设计红线**:表里除向量外,必须同时存原始文本块(给 BM25)和能关联论文元数据(给过滤)。若只存向量,将来做 D 和 E 需推倒重来。图表的原图与位置信息必须在 PDF 解析当下采集,事后无法补回。

## 九、生成层(RAG 的 "G")

前面所有检索工作(A–F、查询理解)最终都为这一步喂料:检索找到"相关内容",生成层把它变成"用户要的答案"。这是用户**真正看到**的东西。

对学术 + 量化场景,本层有一条压倒一切的原则,先立在最前面。

### 9.1 最高原则:宁可明确说没有,不可无据地生成

学术与量化场景里,一个编造的结论、一个错误的引用,远比"没找到"更糟 —— 量化研究中一个瞎编的因子结论会直接导致错误决策、造成损失。

模型有"填空冲动":给它空的或不足的 context 却要求回答,它会从自身参数记忆里编造内容补齐,这是最危险的幻觉来源。因此:

- **检索为空 → 根本不进生成**,直接返回"未找到相关论文 / 未找到足够依据",不给模型对着空白编造的机会。
- **一个诚实的"没有"永远优于一个无据的答案。** 这条原则压在所有"如何生成得更好"的考量之上。

### 9.2 context 组织:依检索路而定(depend on A–F)

context 如何组织**不是一套固定规则,而取决于走了哪条检索路** —— 因为不同层级检索回来的东西形状本就不同。而检索路又由查询意图(第七节)决定。从入口到出口是一条被意图串起的链:

**意图 → 路由到检索路 → 检索路决定 context 组织 → context 形状决定答案形态。**

| 检索路 | context 组织方式 | 典型答案形态 |
|------|------|------|
| **A 摘要** | 多篇论文列表(宽而浅),每篇标题 + 摘要 + 出处 | 论文清单 + 简介 |
| **B 段落** | 段落原文 + 出处;同篇的连续块按原文顺序排,勿打乱 | 段落式回答 + 引用 |
| **C 章节** | 按论文分组(论文 × 章节的矩阵),便于逐篇对比 | 结构化对比(常带表格) |
| **D 过滤** | 不改变形状,只在 A/B/C 之前筛掉一批,组织仍随底层 | 随底层 |
| **E 混合** | rerank 精排后的块,排序更可信,可更激进只取 top 几 | 高质量段落回答 |
| **F 图谱** | 带引用关系的论文集合(图结构),关系本身即内容 | 演化脉络叙述 |

统一要点(所有路共通):按相关性排序、每个单元**必须携带完整出处元数据**(论文标题 / arxiv_id / 章节)—— 这是 9.3 引用标注的基础;注意 lost-in-the-middle 效应(模型更关注 prompt 首尾),重要内容勿全埋中间;控制单次进入生成的单元数量,避免稀释重点与浪费 token。

### 9.3 引用标注:严格、可验证

学术 RAG 与普通聊天机器人的分水岭。本项目采用**最严格档**:

- **句子级标注**:答案中每个论断标注其依据的来源块(如句末 `[1]`,`[1]` 对应某篇论文某段)。
- **附可验证原文片段**:标注不止给编号,还能让用户点开核对对应的原文块 —— 量化研究中结论关乎决策,必须可追溯、可验证,不能只是"看起来有出处"。
- 检索层已为每个块携带出处元数据(9.2),生成时引用对应编号即可,顺势而成。

### 9.4 防幻觉手段(可叠加)

- **prompt 强约束**:明确指令"只依据提供的内容回答,内容中没有的就说找不到,不得用自身知识补充"。
- **空结果拦截**:即 9.1,检索为空不进生成。
- **答案可溯源性校验(进阶)**:生成后校验每个论断是否真有对应检索块支撑,无支撑的论断标记或剔除。较重,作为增强。

### 9.5 答案形态随意图而变

呼应 9.2 的表格,答案形态跟着检索路 / 意图走:找论文(A)给论文清单;问细节(B)给段落式回答 + 引用;综合对比(C)给结构化对比 / 表格;脉络(F)给演化叙述。生成层需按意图选择相应的输出模板。

### 9.6 生成用模型

- embedding 模型与生成 LLM 是**两个不同的东西**:检索用 embedding,生成需要会读写、会推理的对话模型(GPT-4 类)。
- 采用 OpenAI 对话模型,与 embedding 共用同一套 API 凭证。具体型号按成本 / 质量权衡选择。

### 9.7 与第三部分的衔接

生成层与第三部分的"论文问答 Agent"是同一件事的两个深度:本层解决"检索到内容后如何生成带引用、防幻觉的答案";第三部分在此之上做成完整的多步问答 / 趋势分析 / 因子提取 Agent。两者边界是软的 —— 生成层是通往第三部分的桥。

## 十、一个不成熟的设想:Agent 学习笔记 / 动态知识图谱

> 本节是初步设想,尚不成熟,记录备议。它是 F 层(GraphRAG)的一个扩展方向。

设想:让 Agent 把研究过程沉淀成一张**动态知识图谱**作为"学习笔记",把搜索、阅读过的论文与主题串联起来。例如 `Factor → Fourier → A股` 是图中的一条路径,而 `Factor`、`Fourier`、`A股` 各自又是独立节点,能与其他节点组成别的路径与组合。

关键洞察:这里其实融合了**两种来源不同的图**——

- **客观引用图(即 F 层)**:边是"谁引用谁",来自 Semantic Scholar 的引用数据,反映学术界**已有的**知识脉络。
- **主观研究轨迹图(本设想新增)**:边是"我们研究时把它们联系在了一起",来自团队自身的检索 / 阅读行为,反映**本团队**关注的主题如何关联、如何演进。

两者叠加,节点可以是论文、也可以是主题概念(Factor、Fourier 这类);边可以是客观引用、也可以是主观关联。对量化团队而言,后者(研究轨迹)甚至更有价值 —— "我们探索过哪些主题组合、哪条路走通了"是团队独有、逐步积累的资产,他人无法复制。

尚待厘清的设计点(均为真问题,故标"不成熟"):

- **节点定义**:只用论文,还是也纳入主题概念节点?两者如何共存?
- **概念抽取**:`Factor` 这类主题词如何从查询 / 论文中自动提取?
- **边的建立与加权**:搜索先后自动连边,还是需人工确认?边如何加权(共现次数 / 时间邻近)?
- **动态更新与可视化**:图如何随研究进展增长?如何呈现为可交互的"学习笔记"?

定位:F 层的进阶延伸,团队特色资产方向,非近期实现目标,先记录设想。

## 十一、评估体系

前面每一块都做了大量"我认为这样更好"的设计判断(递归切、512 token、加 reranker、树状分解封顶 6 个……)。评估体系的作用是**把这些判断从"听起来合理"变成"有数据证明"**——这是 demo 与研究的分界线,也是向团队 / 学长证明工作价值的客观依据。

它还有一个日常价值:A–F 每加一层、每调一个参数,都需要一把尺子量"到底变好还是变坏"。没有尺子,调参就是黑暗里凭感觉。

核心结构:**RAG 有两半(检索 + 生成),要分开评,因为它们坏的方式不一样。**

### 11.1 测试集:一切评估的前提

需要一批"标准题 + 标准答案",例如:问题"哪篇论文提出了 ReAct 方法?" → 标准答案"应检索到 arxiv_id 2210.03629"。

**构建方式:LLM 批量生成 + 人工筛选。** 用 LLM 读论文自动生成"该论文能回答的问题"(该论文即标准答案),批量产出候选;团队再筛选、修正出靠谱的若干条。**几十到上百条高质量测试题即足以开始量化**,不必一开始追求上千条。

### 11.2 检索评估:量"找得准不准"

评估检索层是否把该找的内容找回、排得够前。采用标准指标(现成,无需自创):

- **Recall@k(召回率)**:该找到的是否出现在前 k 个结果里 —— 衡量"漏没漏"。
- **MRR(平均倒数排名)**:找到了,但排第 1 还是第 10 —— 衡量"排得好不好"。
- (可选 NDCG、Hit Rate 作补充)

**主指标:Recall@k + MRR。** 核心用途是做 **A/B 对比实验**,量化每个设计决策的真实收益:

- 纯向量 vs 向量 + BM25(E 层):Recall 提升多少?
- 加 reranker vs 不加:MRR / NDCG 提升多少?
- 512 token vs 256 token 切块:哪个召回更高?

### 11.3 生成评估:量"答得好不好"

答案是自由文本、无唯一正确形式,比检索难量化。评估维度:

- **忠实度(faithfulness)**:答案是否真正基于检索内容、有无幻觉。**对本项目最重要**,直接对应生成层"绝不无据生成"的红线。
- **相关性(relevance)**:是否答到点子上。
- **引用正确性**:标注的引用是否真的支撑对应论断。

**方法:LLM-as-judge(用 LLM 当评委)为主**,让另一个 LLM 按标准 rubric 给答案打分。可规模化、快,是当前学术界与工业界的主流做法。可直接使用现成的 RAG 评估框架(如 RAGAS),不必自造轮子。重点评**忠实度** —— 它是本项目的命门。

### 11.4 忠实度做成硬门槛

忠实度不只是一个观测指标,而是一道**可执行的关卡**:

- 生成层"设计上不让它编",评估层"量它到底编没编",二者是同一件事的两面。
- **忠实度低于设定阈值的答案,系统直接拦下、不返回给用户**(退化为"未找到足够依据"),将"反幻觉"从原则变成硬性关卡。

### 11.5 评估融入日常:当回归测试用

评估最忌"做一次、写进报告、再不跑"。真正有用的评估是**持续的**:

- **回归测试思维**:做成可一键运行的评估脚本,每次改参数 / 加新层都跑一遍,防止"改好 A 却弄坏 B"。
- **结果留档**:每次评估结果存下,形成系统质量随时间的变化曲线。既是质量守门员,也是汇报的数据来源。

### 11.6 为什么不能单靠 LLM 评委——三个坑

LLM-as-judge 好用,但**单独使用有三个已知的坑**,必须配合防护措施,否则评分会系统性失真、不可信:

**坑一:偏见。** LLM 评委会偏爱长答案、偏爱自己风格的措辞、受选项顺序影响(A/B 对比时偏袒排前面的)。
→ **防护:rubric 要具体、可判定。** 不问"这个答案好不好"(任其发挥偏见),要问"答案中每个论断是否都能在给定内容里找到依据?有几处找不到?"(客观、可核对)。把主观评价拆成客观小问题,偏见无处施展。

**坑二:自我偏袒。** 若生成与评委是同一模型(如都用 GPT-4),评委会给"自己风格"的答案打高分(self-preference bias)。
→ **防护:评委尽量用与生成不同的模型**,或至少明确意识到此偏差。**本项目尤需注意** —— 生成与评估很可能都用 OpenAI,极易踩中。

**坑三:评委本身也会错。** LLM 对忠实度的判断自身有出错率,完全当真理会被其错误误导。
→ **防护:用少量人工评分校准 LLM 评委。** 让人工与 LLM 各评同一批(如 20 条),看一致率。吻合度高,才可信任它规模化评剩余数百条;吻合度差,说明评委 prompt 有问题需修。**此步是"让 LLM 评委本身变得可信"的关键** —— 有了它,才能理直气壮地说"我们的 LLM 评委与人工判断吻合度 90%,故其分可信",而非"我让模型打了个分"。

> 三个防护措施成本都不高,却是"随便用 LLM 打分"与"严谨地用 LLM 评估"的区别,也是研究严谨性的直接体现。

## 十二、对外接口

系统做出来了,别人怎么用它?关键认识:**"别人"有三类,各要不同形态的接口,但共享同一个后端引擎**。这与"统一底座、多种粒度"一脉相承 —— 统一检索引擎,多种接口门面。

```
        ┌─────────────────────────────────┐
        │  统一后端引擎(查询理解→检索→生成) │
        └─────────────────────────────────┘
                        │
         ┌──────────────┼──────────────┐
      前端界面        REST API        MCP 接口
      (给人)         (给程序/Agent)   (给 AI 助手)
```

### 12.1 三类用户

- **社团成员(人)**:想搜论文、问问题,要**能点能看的界面**,不会去调 API。
- **量化程序 / 其他 Agent(机器)**:如因子研究流程想自动查"有无相关论文",要**能被代码调用的接口**。
- **AI 助手(如 Claude)**:想把论文库当工具调用,要 **MCP 这类给 AI 用的标准接口**(呼应招募技术栈中的 MCP / tool calling)。

### 12.2 核心 API 层(所有门面的地基)

用 FastAPI(已在技术栈)建统一 API 服务,把"查询理解 → 检索 → 生成"封装为可调用端点。三种门面都基于它。端点草案:

```
POST /search          纯检索,不生成
  输入: { query: str, granularity?: "abstract"|"chunk"|"section",
          filters?: {year_min, citation_min, has_code...}, top_k?: int }
  输出: { results: [ {text, paper_id, arxiv_id, title, section,
                      score, source_url} ], query_understanding: {...} }

POST /ask             完整 RAG(检索 + 生成)
  输入: { query: str, options?: {granularity, filters, strict_citation} }
  输出: { answer: str, citations: [ {marker, paper_id, arxiv_id,
                                     quote, source_url} ],
          sources: [...], faithfulness_score: float,
          status: "ok"|"no_result" }

GET  /paper/{arxiv_id}    取论文详情
  输出: { metadata, abstract, sections?, figures: [...], tables: [...] }

GET  /paper/{arxiv_id}/figure/{fig_id}    取单个图表(供悬浮插件)
  输出: { image_url|structured_data, caption, position, related_text_refs }
```

> `/ask` 的 `status: "no_result"` 对应生成层 9.1 的空结果拦截;`faithfulness_score` 对应 11.4 的忠实度门槛。出处元数据一路带到输出,不中途丢弃(呼应 12.5)。

### 12.3 前端界面(给人)—— UI 技术选型待定

**待定:UI 技术栈(明天与学长共同拍板)**

| 方案 | 优势 | 劣势 | 适用 |
|------|------|------|------|
| **Streamlit / Gradio** | 纯 Python,团队零前端成本,当天出 demo | 复杂交互(悬浮、图谱)受限 | 快速验证 / 内部使用 |
| **React(+ Vue 备选)** | 功能上限高,复杂交互无碍 | 需额外前端技术栈 | 完整产品形态 |

建议(待议):**两层并行** —— 快速验证 / 内部先用 Streamlit(零成本、见效快);面向完整形态(悬浮插件、知识图谱可视化)规划 React,因其复杂交互 Streamlit 撑不住。若团队已有 React 能力,可直接走 React。

界面的完整形态包含:

- **搜索 + 结果列表**:基础,任何方案都做。
- **论文详情页**:展示全文、图表、元数据。
- **悬浮插件**(见 5.6):前端渲染论文正文时,把"图3 / Table 2"这类交叉引用渲染成**可交互元素**,悬浮时调用 `/paper/{id}/figure/{fig_id}` 拉取图表数据,浮出原图 + caption + 结论。**它不是独立程序,是阅读界面里的一个前端组件。**
- **知识图谱可视化**(见第十节设想):**待定** —— 用图网络库呈现动态知识图谱。候选:Cytoscape.js / vis.js(专画网络图,较易上手)或 D3.js(最灵活但最难)。归入 React 完整版。

### 12.4 MCP 接口(给 AI 助手)—— 招募技术栈重点

MCP(Model Context Protocol)让 AI 助手把论文库当"工具"调用。做成后,用户在 Claude 等助手里说"在 XX 社团论文库里找 agent 相关论文",助手会自动调用本接口。

- 本质:把 12.2 的 API 端点**包装成 MCP 规范的工具描述**(工具名、参数 schema、说明),让 AI 能理解与调用。
- 复用核心 API,只是换一层符合 MCP 规范的外壳。
- 属招募文案明确方向(MCP / tool calling / A2A),非可选项。

### 12.5 统一返回格式(贯穿三门面)

三门面共享后端,则返回数据必须**结构统一、自带出处**:

- 给人的界面:展示"这段来自哪篇论文哪节"。
- 给程序的接口:能解析出处做后续处理。
- 给 AI 的 MCP:让 AI 知道每条信息来源以正确标引用。

定义统一返回结构(结果 + 出处元数据 + 评分),三门面共用,仅渲染方式不同。这与生成层"每个块带出处元数据"一脉相承 —— **出处信息一路带到接口输出,不中途丢失**。

## 十三、工程运维(流水线调度 + 成本性能)

前面各节讲"系统有什么能力",本节讲**如何让这些能力自动、稳定、省钱地长期跑下去**。它不增加新功能,但决定这套系统是"能演示的 demo"还是"能天天用的产品"。

### 13.1 完整数据流水线

第一部分(爬取→增强→入库)与第二部分(解析→切块→embedding→入库)现为割裂的脚本,实则是一条链:

```
爬取 → 增强 → PDF解析 → 切块 → embedding → 入库 →(可被检索)
```

目标:让这条链**自动、按序、按各自节奏运行**,而非手动逐个触发。

### 13.2 分节奏编排(不是一条线跑到尾)

各环节节奏不同(呼应第六节 embedding 调度),故拆成多条:

- **每日轻量线**:爬取 → 增强 → 摘要 embedding(A)。新论文天天有,高频。
- **低频 / 按需重线**:PDF 解析 → 切块 → 全文 embedding(B/C)。量大成本高,不必每日全量。

### 13.3 失败处理:全链路状态字段驱动

一条链在中间某步失败,前面成果不能丢、不能整条重跑。延续第一部分已验证的**状态字段驱动**模式:

- 每个阶段一个状态位:已爬取 / 已增强 / 已解析 / 已 embedding(如 `enriched`、`embedded`)。
- 每个环节只处理"尚未做到这一步"的记录,失败的保持未完成状态、下次自动重试。
- 流水线因此**可中断、可续跑、可自愈**。此模式从第一部分一以贯之到全链路。

### 13.4 部署平台:GitHub Actions 与 Railway 各管一段

**关键区分:二者不是竞品,本质不同。**

- **GitHub Actions = 定时触发器("闹钟")**:到点起临时环境跑脚本、跑完销毁。适合短时、周期性任务,不常驻,免费额度够用,与代码仓库天然集成。
- **Railway = 托管服务器 / 容器("一直开着的电脑")**:程序 24 小时常驻运行。适合常驻服务与长时重活,按用量付费。

分工:

| 组件 | 平台 | 理由 |
|------|------|------|
| 每日轻量线(爬取/增强/摘要embedding) | **GitHub Actions cron** | 定时、短时、免费、契合"闹钟"模式 |
| 全文重线(解析/切块/全文embedding) | **GitHub Actions**(能在时限内跑完)或 **Railway**(量大) | 批量任务,不需常驻;视单次耗时定 |
| FastAPI + MCP 接口 | **Railway(必须)** | 需 24 小时在线等待请求,Actions 不常驻、做不了 |

### 13.5 【前置待办】数据库上云

**当前 PostgreSQL 跑在本地开发机(`brew install` 那套)。本地库有致命限制:开发机关机,GitHub Actions 与 Railway 都无法连接它。**

因此自动化的地基是:**数据库必须从本地迁移上云**。否则定时任务连不上库,整个自动化无从谈起。

- 候选:Railway Postgres(与接口同平台)/ Supabase / Neon(专门的托管 Postgres,均支持 pgvector)。
- 定位:这是比"Railway 还是 GitHub"更前置的决定 —— 本地这套目前只是**开发环境**,上线必须先解决数据库上云。

### 13.6 成本与性能

- **缓存(Redis,已在技术栈)**:同一查询不重复跑完整检索 + 生成。缓存热门查询结果与 embedding,重复查询直接命中,省钱省时。
- **批处理**:见第六节 —— 按 token 动态攒批;大批量用 OpenAI Batch API(异步半价)。归入成本控制整体视角。
- **pgvector 索引调优**:向量检索速度取决于索引。`ivfflat` 的簇数参数平衡速度与精度。**先用默认参数跑通,数据量(几十万向量)上来后再调 `ivfflat` 或升级到 HNSW 索引**。不提前优化,但明确旋钮所在。

### 13.7 部署全景

```
每日轻量线(爬取/增强/摘要embedding)  → GitHub Actions cron(免费)
全文重线(解析/切块/全文embedding)    → GitHub Actions 或 Railway(视量)
FastAPI + MCP 接口(需常驻)           → Railway 必须
数据库                                → 【前置】本地上云:Railway PG / Supabase / Neon
缓存                                  → Redis
```

## 十四、讨论点

1. **优先级确认**:A–E 是否都要做?F(GraphRAG)是本阶段目标还是长期规划?
2. **PDF 解析方案**:普通解析(PyMuPDF)是否够用,还是直接上 GROBID 做学术结构化解析?
3. **Embedding 成本与选型**:先用 OpenAI API 跑通,还是一开始就上本地 BGE?
4. **章节识别(C 层)**:靠 PDF 解析工具自带的结构,还是需要额外规则/模型识别?
5. **图表架构投入**:表格结构化(档位2)与图表库、交叉引用解析是第一版就做,还是先跑通纯文本检索再回补?(注:原图与位置信息须在解析当下采集,事后补不回)
6. **B/C 全文 embedding 的调度**:全量做还是按需 / 按热度做?(A 随爬虫高频增量已定;B/C 量大成本高,频率策略待定)
7. **查询理解的投入**:第一版是否只做改写 + 双语扩展?树状分解的三道闸(判断/2层/6叶子)参数是否认可?
8. **接口与 UI 选型(待共同拍板)**:UI 走 Streamlit 快速版还是 React 完整版,或两层并行?知识图谱可视化用 Cytoscape/vis.js 还是 D3?MCP 接口优先级?
9. **生成层的引用严格度**:采用最严格档(句子级标注 + 可验证原文片段)已定;"空结果拦截、宁可说没有"作为最高原则是否共识?
10. **评估体系投入**:测试集规模(几十/上百条)?忠实度硬门槛的阈值定多少?人工校准投入多少人力?
11. **与量化研究的结合**:是否需要"因子信号提取"这类量化特供的检索/抽取能力,作为 D 或 E 之上的应用层?
12. **知识图谱设想(第十节)**:客观引用 + 主观研究轨迹的融合图,是否值得作为长期特色方向立项?
13. **【前置】数据库上云**:本地 PostgreSQL 须迁云才能跑自动化,选 Railway PG / Supabase / Neon?(比部署平台更前置)
14. **部署平台分工**:轻量线 GitHub Actions + 常驻接口 Railway 的分工是否认可?全文重线放哪?
