# AI 论文 RAG 系统 —— 第二部分规划

> 检索层设计与落地路线
> AI Agent 组 · 第二阶段
> 更新日期:2026-07

---

## 一、背景与定位

第一部分(数据采集 + 增强)已完成并跑通:

- arXiv API 抓取 AI / Agent 方向最新论文,入库 PostgreSQL
- 从 Semantic Scholar、Hugging Face Papers 两个来源异步补充 arXiv 没有的字段(引用数、影响力引用数、参考文献数、社区点赞数等)
- Papers with Code 已在 2025 年 7 月被 Meta 关停,已从数据源中移除;其代码链接数据后续可从 Hugging Face 的 `linkedModels` / `linkedSpaces` 补回

第二部分的目标:**在已有论文数据之上,建立一套完整的检索系统**,让用户能够按不同需求从论文库中精准获取内容。

核心设计原则:**统一底座,分层填充,共用一张表一套接口**。检索能力分为 A–F 六个层级,层层叠加,不是六套独立系统。

---

## 二、检索能力的六个层级(A–F)

| 层级 | 能力 | 解决的问题 | 依赖的数据 |
|------|------|-----------|-----------|
| **A** | 摘要级语义检索 | "有没有讲 X 的论文" —— 定位到整篇 | 摘要(第一部分已有) |
| **B** | 段落级语义检索 | "这篇论文的方法具体怎么做" —— 定位到段落 | PDF 全文 |
| **C** | 章节级语义检索 | "跨多篇论文,综合某个技术方向" —— 按章节抽取 | PDF 全文 + 章节结构 |
| **D** | 元数据过滤检索 | "2024 年后、引用 > 50、带代码的论文里……" | 第一部分的增强字段 |
| **E** | 混合检索 + 重排 | 补语义检索短板(精确术语、论文名) | 原始文本 + BM25 + reranker |
| **F** | 引用网络检索(GraphRAG) | "这个方法从哪演化来、被谁改进" —— 沿引用链推理 | 引用关系图 |

### 逐层说明

**A —— 摘要级语义检索**
最简单的一层。摘要在第一部分已经抓到,只需做 embedding 存入向量库。粒度是"一篇论文 = 一个向量"。适合"找论文"的需求。

**B —— 段落级语义检索**
需要下载 PDF、解析全文、切成段落块,每块单独 embedding。粒度是"一篇论文 = 几十个块"。适合"问某篇论文的具体细节"。

**C —— 章节级语义检索**
在 B 的基础上,识别每个块属于哪个章节(intro / method / experiment / conclusion),打上标签。适合"跨多篇论文综合某个特定方面"(如只对比各篇的方法部分)。

**D —— 元数据过滤检索**
在语义检索之上叠加结构化字段过滤。用第一部分攒下的 `published`、`citation_count`、`code_urls` 等字段,先缩小候选范围再做语义匹配。第一部分数据增强的价值在这一层兑现。

**E —— 混合检索 + 重排**
A–D 只用了"向量检索"一条腿。E 加上第二条腿(BM25 关键词检索),两路结果用 RRF 融合,再用 reranker(如 `bge-reranker-v2-m3`)精排。补足向量检索对精确术语、专有名词、论文名的短板。**这一层是检索质量的分水岭。**

**F —— 引用网络检索(GraphRAG)**
最复杂,也是天花板。把论文当成互相连接的网络而非孤立文本块,用引用关系做图上推理。工程量最大(需要图结构),但正是"最前沿"方向所指。可作为长期目标。

### 层级依赖关系

```
A  摘要级语义检索        ┐
B  段落级语义检索        ├─ 语义相似度(一条腿)
C  章节级语义检索        ┘
D  + 元数据字段过滤      ← 用第一部分攒的字段
E  + BM25 + reranker     ← 检索质量分水岭
F  + 引用网络 / GraphRAG ← 最前沿,天花板
```

越往下越依赖前面每一步攒的数据:D 靠增强字段,F 靠引用关系。这也说明第一部分抓取 citation / reference / code 字段并非多余,而是为 D 和 F 铺路。

## 三、技术选型(初步)

| 环节 | 选择 | 理由 | 备选 |
|------|------|------|------|
| 向量库 | PostgreSQL + pgvector | 复用已有数据库,一库同时管结构化数据与向量 | Qdrant / Milvus(规模大后再评估) |
| Embedding 模型 | OpenAI `text-embedding-3-small` | 已有可用 API key;成本极低(约 $0.02 / 百万 token);跑通快 | BGE 系列(需本地部署,后续可省钱/离线时切换) |
| PDF 解析 | 待第二步选型 | —— | PyMuPDF / GROBID(学术论文结构化解析) |
| Reranker | `bge-reranker-v2-m3` | 中英文混合,开源可本地跑 | LLM Reranker(效果更好但需 GPU) |

> 说明:选型原则是"当前规模和阶段下,工程复杂度最低、效果足够好",非绝对最优。规模上来后 D/E/F 相关组件均可重新评估。

## 四、落地路线

底座一次设计到位(支持到 E),能力按 A→B→C→D→E→F 逐层填充,避免返工。

### 里程碑

**第一步:搭底座 + 填 A**
- 建 pgvector 扩展与统一检索表(设计见第五节)
- 摘要 embedding 入库
- 跑通"embedding → 入库 → 检索"最小链路
- 用已有摘要数据,不碰 PDF,最快见效

**第二步:填 B**
- PDF 下载 + 解析 + 段落切块
- 切块 embedding 存入同表,`granularity = chunk`
- 专注解决 PDF 解析的坑(双栏、公式、表格、扫描版)

**第三步:填 C**
- 解析时识别章节结构,给块打 `section` 标签
- 在 B 的解析代码上扩展,非重写

**第四步:填 D**
- 检索接口叠加元数据过滤(关联第一部分的 papers 表)

**第五步:填 E**
- 接入 BM25 关键词检索
- RRF 融合向量 + BM25 结果
- reranker 精排
- 检索质量的关键提升

**第六步(长期):填 F**
- 建引用关系图(单独的边表或图数据库)
- 图上检索 / 推理
- 最前沿方向

## 五、统一表设计要点

一张 pgvector 表覆盖 A–E,靠字段区分粒度与来源。关键字段:

- **`granularity`** —— 区分粒度:`abstract` / `chunk` / `section`,支撑 A/B/C
- **原始文本字段** —— 不只存向量,同时存原文块。**E 层 BM25 需要原始文本,不是向量**;只存向量则 E 无法实现
- **论文元数据关联** —— 关联第一部分的 `papers` 表(或冗余关键字段),支撑 D 的过滤
- **embedding 向量字段** —— `vector(1536)`,加 `ivfflat` 索引
- **章节标签字段** —— 支撑 C

> F 层的引用图需要额外的边表,建立在同一批论文数据上,不与本表冲突,后续单独设计。

**设计红线**:表里除向量外,必须同时存原始文本块(给 BM25)和能关联论文元数据(给过滤)。若只存向量,将来做 D 和 E 需推倒重来。

## 六、开会讨论点

1. **优先级确认**:A–E 是否都要做?F(GraphRAG)是本阶段目标还是长期规划?
2. **PDF 解析方案**:普通解析(PyMuPDF)是否够用,还是直接上 GROBID 做学术结构化解析?
3. **Embedding 成本与选型**:先用 OpenAI API 跑通,还是一开始就上本地 BGE?
4. **章节识别(C 层)**:靠 PDF 解析工具自带的结构,还是需要额外规则/模型识别?
5. **检索接口形态**:最终对外是 FastAPI 接口 / MCP tool / 内部 CLI?(呼应第一部分技术栈)
6. **与量化研究的结合**:是否需要"因子信号提取"这类量化特供的检索/抽取能力,作为 D 或 E 之上的应用层?

## 附:与第一部分的衔接

第一部分产出的每一类数据,在第二部分都有明确用途:

- 摘要 → A 层直接使用
- PDF 链接 → B/C 层下载解析的入口
- 引用数 / 影响力 / 点赞数 → D 层过滤与排序加权
- 参考文献 / 引用关系 → F 层构图的基础
- HF 的 `ai_summary` / `ai_keywords` / `linkedModels` → 未来标签与代码链接的来源
